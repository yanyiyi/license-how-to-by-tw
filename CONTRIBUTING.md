# Contributing Guide

謝謝你願意來看這個專案，也歡迎一起貢獻。

這個專案主要想整理一些和創用 CC、開放授權、網站內容使用、案例回覆、授權標示有關的實務經驗。
很多事情不是只有「可不可以」這麼簡單，而是要看素材來源、授權條款、使用方式、顯名標示、個資肖像、第三方素材等條件。這些經驗如果能整理下來，後面的人就可以少踩一點坑。

不管你是想修錯字、補案例、分享某個網站怎麼判斷授權、補一段比較好的說明，或是送 PR 改工具設定，都很歡迎。

## 授權方式

這個專案採用兩種授權：

- 程式碼、scripts、設定檔、範例程式：MIT License
- 文件、案例整理、問答文字、教學說明、標示範例：CC BY 4.0

也就是說：

> 如果你貢獻的是程式碼，會用 MIT License。
> 如果你貢獻的是文字、文件或案例，會用 CC BY 4.0。

請參考：

- `LICENSE`：MIT License
- `LICENSE-docs.md`：CC BY 4.0

如果內容裡有引用第三方素材、網站內容、別人的文章或圖片，請另外標示來源與授權。
第三方素材不會因為出現在這個專案裡，就自動變成本專案的 MIT 或 CC BY 4.0 授權。

## 不採用 CLA

這個專案不要求簽 CLA。

我希望大家可以比較輕鬆地分享經驗、開 issue、送 PR，不要一開始就被一份法律文件擋在門口。

不過，還是請確認幾件事：

- 你有權利提交這份內容
- 這份內容可以依本專案的授權釋出
- 你沒有直接貼上未授權的大量內容
- 你沒有提交個資、機密資料、token、API key 或不該公開的內容
- 如果有第三方內容，已經清楚標示來源與授權狀態

## DCO / Signed-off-by

這個專案採用 DCO，也就是 Developer Certificate of Origin 的精神。

DCO 不是 CLA。
它比較像是你在 commit 裡簡單聲明：

> 我有權利提交這份貢獻，也同意它依照這個專案的授權方式釋出。

送 PR 時，請在 commit message 加上：

```text
Signed-off-by: Your Name <your.email@example.com>
```

用 git commit 時可以這樣做：

```bash
git commit -s -m "Add CC BY attribution example"
```

如果已經 commit 但忘記加，可以用：

```bash
git commit --amend -s
```

第一次忘記也沒關係，維護者可以協助提醒。這不是要嚇跑大家，只是希望貢獻來源比較清楚。

## 歡迎貢獻什麼？

很歡迎這些內容：

- 修錯字、修格式、補連結
- 補充 CC 授權問答
- 補充授權標示範例
- 補充網站內容怎麼判斷授權
- 補充案例拆解
- 補充回覆模板或常見問法
- 補充第三方素材、個資、肖像、Logo、AI 使用等注意事項
- 補充工具設定、prompt、skill、checklist
- 回報文件過時、連結失效或說明不清楚的地方

不用一開始就寫得很完整。
如果還不知道怎麼整理，可以先開 issue 或 discussion，大家一起把它整理成比較好用的版本。

## 分享案例時，請先去識別化

這個專案很歡迎案例，但不希望直接貼原始信件或內部文件。

請盡量把案例改寫成「模式」：

```markdown
## 案例情境

某使用者想使用一張採 CC BY-NC 授權的圖片，放在商業網站的介紹頁中。

## 可能涉及的問題

- CC BY-NC 是否允許商業網站使用？
- 是否需要額外授權？
- 圖片要怎麼標示來源？
- 如果只作為年表或介紹素材，是否仍可能被認定為商業使用？

## 建議方向

先確認網站性質與使用目的。若屬商業使用，建議不要直接依 CC BY-NC 使用，而是回頭洽權利人或平台窗口取得額外授權。若是非商業使用，也仍需保留合理的姓名標示與來源資訊。
```

請避免放入：

- 真實姓名
- 電話、Email、地址
- 未公開信件
- 合約全文
- 內部討論
- 還沒取得同意公開的諮詢內容
- 可識別特定人的敏感細節

如果案例真的很有參考價值，可以改成假名、概括描述或最小化範例。

## 分享網站處理經驗

也很歡迎大家分享「某個網站要怎麼看、怎麼判斷、怎麼處理」。

可以包含：

```markdown
## 網站或資料來源

- 網站名稱：
- 網址：
- 資料類型：
  - 例如：文章、圖片、PDF、表格、影音、metadata
- 想做什麼：
  - 例如：引用、摘要、整理、擷取欄位、製作標示範例

## 授權與使用條件

- 網站有沒有明確授權？
- 是否有 CC 授權？
- 是否有政府開放資料授權？
- 是否有使用條款？
- 是否限制商業使用、重製、下載或爬取？

## 技術狀況

- 是否需要 JavaScript？
- 是否有登入、cookie、驗證？
- 是否是 PDF、圖片表格或動態載入？
- 是否有 robots.txt 或 rate limit 要注意？

## 風險提醒

- 是否涉及著作權？
- 是否涉及個資、肖像或隱私？
- 是否涉及 Logo 或商標？
- 是否有第三方素材混在裡面？
- 是否只適合摘要，不適合全文複製？

## 建議做法

- 可以怎麼引用？
- 可以怎麼標示？
- 哪些做法比較穩？
- 哪些做法不建議？
```

重點不是鼓勵大家亂抓資料，而是把「怎麼判斷比較穩妥」的經驗留下來。

## 標示來源很重要

這個專案很在意來源標示。

如果你補充資料，請盡量保留：

- 作品名稱
- 作者或權利人
- 來源網址
- 授權條款
- 授權版本
- 是否有修改
- 查閱日期
- 是否有第三方素材或例外註記

簡單範例：

```text
〈作品名稱〉，作者：作者名稱，來源：網址，授權：CC BY 4.0，已修改。
```

如果是第三方素材，請不要讓讀者誤會它也跟著本專案一起採 CC BY 4.0。

可以這樣寫：

```text
除另有標示者外，本文件內容採 CC BY 4.0 授權釋出。文中第三方素材仍依其原授權或法律狀態處理。
```

## AI 輔助內容

可以使用 AI 協助整理、潤稿、摘要或檢查格式。
但送出 PR 前，請自己確認內容正確、來源清楚、授權沒有問題。

如果 PR 有大量使用 AI 協助，可以在說明裡簡單註明：

```text
本 PR 部分內容曾使用 AI 工具協助整理，已由貢獻者人工確認與修訂。
```

請不要把未公開信件、合約、個資、內部文件或敏感案例直接丟進第三方 AI 工具後再提交。

## Pull Request 怎麼送？

大致流程：

1. Fork 這個專案
2. 開一個新的 branch
3. 修改內容
4. 確認格式、連結、來源標示
5. 確認沒有個資、token、密碼或不該公開的內容
6. commit 時加上 `Signed-off-by`
7. 送 Pull Request

PR 說明可以簡單寫：

```markdown
## 這個 PR 做了什麼？

-

## 為什麼要改？

-

## 內容類型

- [ ] 文件
- [ ] 案例
- [ ] 授權標示範例
- [ ] 網站處理經驗
- [ ] 程式碼
- [ ] 其他：

## 授權與來源確認

- [ ] 我確認本 PR 中由我貢獻的程式碼可依 MIT License 釋出
- [ ] 我確認本 PR 中由我貢獻的文件內容可依 CC BY 4.0 釋出
- [ ] 如果有第三方內容，我已標示來源與授權
- [ ] 本 PR 不包含不應公開的個資、機密、帳密或 token

## 是否需要特別討論？

-
```

## 文件風格

這個專案的文件，希望盡量白話、清楚、可實作。

建議寫法：

- 先說明情境
- 再拆解可能涉及的問題
- 不要一開始就直接下結論
- 可以給建議，但要保留個案判斷空間
- 有標示範例更好
- 有不確定的地方，就明講不確定
- 儘量使用繁體中文與臺灣常用語

可以參考這種結構：

```markdown
## 問題理解

這個問題可能涉及幾個層面：

1. 著作權
2. CC 授權條件
3. 第三方素材
4. 個資或肖像
5. 網站使用條款

## 初步判斷

依目前資訊，較穩妥的理解是……

## 建議做法

1.
2.
3.

## 注意事項

以上是一般性整理，仍需要依實際素材來源、授權條款與使用方式判斷。
```

## 請避免提交

請不要提交：

- 未授權的大量全文轉載
- 未去識別化的諮詢信件
- 未公開合約或內部文件
- 個資、肖像、電話、Email、地址
- 帳號、密碼、token、API key
- 未經同意公開的會議紀錄
- 規避登入、驗證、付費牆或技術限制的做法
- 大量自動化請求或可能造成網站負擔的操作
- 無法確認來源或授權狀態的第三方素材
- 攻擊、歧視、騷擾或貶抑他人的內容

如果需要描述敏感情境，請用假資料、摘要或最小化範例。

## Review 原則

維護者會盡量用友善、務實的方式 review。

可能會請你：

- 補來源
- 補授權標示
- 改成比較白話
- 把個案改成通用 pattern
- 移除不適合公開的資訊
- 改用摘要，不要貼全文
- 補上 `Signed-off-by`

PR 不一定會馬上合併。
有些內容可能需要討論、確認或改寫，但這不代表你的貢獻不重要。

## 不確定時，先討論

如果你不確定某個內容能不能放、某個網站能不能處理、某段案例能不能公開，歡迎先開 issue 或 discussion。

很多有用的整理，都是從一句「我遇到一個有點卡的問題」開始的。

謝謝你願意把經驗留下來，讓後面的人有路可走。
